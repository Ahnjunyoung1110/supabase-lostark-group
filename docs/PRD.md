# 로스트아크 레이드 약속 관리 앱 — PRD

## 1. 개요

친구 그룹이 로스트아크 레이드 약속을 잡을 때, 약속 시간을 정해 올리면
멤버들이 참석 여부를 응답하고 그 결과가 자동 집계되는 웹앱.
총무가 수동으로 가능 인원을 종합하던 수고를 없앤다.

## 2. 목표 / 비목표

### 목표

- 시간이 정해진 약속을 올리고, 멤버별 **참석 / 불참 / 미정** 상태를 한눈에 본다.
- 약속 시간 변경(미루기)을 **누구나 제안**할 수 있고, 주최자가 확정한다.
- 일회성 약속과 반복(정기) 약속 모두 지원한다.

### 비목표 (이번 범위 제외)

- 레이드 공략/숙제 관리, 골드 분배, 캐릭터 스펙 관리
- 외부 알림(디스코드 푸시/Webhook 등) — 추후 검토
- 권한 등급/관리자 콘솔 — 친구 그룹 전제, 단순 유지

## 3. 사용자 & 시나리오

- **주최자**: 약속을 올리고 시간 변경을 확정하는 사람 (보통 총무)
- **멤버**: 약속에 참석 여부를 응답하는 친구들

### 핵심 시나리오

1. 주최자가 "토요일 21:00 카멘 하드" 약속을 올린다.
2. 멤버들이 각자 참석 / 불참 / 미정으로 응답한다.
3. 약속 화면에서 상태별 인원과 명단이 실시간 집계된다.
4. 한 멤버가 "22시로 미뤄도 될까?" 시간 변경을 제안한다.
5. 주최자가 제안을 확정하면 시간이 바뀌고 모든 응답이 미정으로 초기화된다 (재확인 유도).

## 4. 기능 요구사항

### 4.1 인증

- 디스코드 소셜 OAuth (Supabase Auth Provider)
- 최초 로그인 시 닉네임(프로필) 자동 생성, 변경 가능
- 비로그인 사용자는 약속 조회/응답 불가 (친구 그룹 전용)

### 4.2 약속 (Event)

| 필드 | 설명 |
|------|------|
| 제목 | 필수 (예: "카멘 하드") |
| 설명 | 선택 |
| 레이드명 | 선택 (예: "카멘", "에키드나") |
| 시작 일시 | 필수 |
| 약속 형태 | 일회성 / 반복 선택 |

- **반복 약속**: 요일+주기 규칙(예: 매주 토 21:00)으로 약속 인스턴스 자동 생성
- **목록 화면**: 다가오는 약속 우선 정렬, 지난 약속 분리 표시
- **상세 화면**: 약속 정보 + 상태별 집계 + 응답 버튼 + 시간 변경 제안 영역

### 4.3 참석 응답 (Response)

- 멤버당 약속별 상태 1개: **참석 / 불참 / 미정**
- 기본값 **미정**, 로그인 상태라면 언제든 변경 가능
- 상세 화면에 상태별 인원 수 + 명단 표시
- 변경은 실시간 반영 (Supabase Realtime)

### 4.4 시간 변경 제안 (Time Proposal)

- **누구나** 새 일시 + 메모를 담아 제안 가능
- 주최자가 제안 목록에서 **확정 / 거절** 선택
- 확정 시: 약속 시간 변경 + 모든 응답을 **미정으로 초기화** (재확인 유도)
- 제안/확정 이력은 약속 상세에 표시

| id | uuid PK | |

| 컬럼 | 타입 | 설명 |

| id | uuid PK | |

### RLS 정책 (친구 그룹 전제 — 인증 사용자 폭넓게 허용)

| 작업 | 허용 대상 |
|------|-----------|
| events / responses / proposals 읽기 | 로그인 사용자 전체 |
| 본인 response upsert | `user_id = auth.uid()` |
| proposal 생성 | 로그인 사용자 전체 |

## 6. 화면 구성

```
/                   랜딩 — 로그인 유도 (기존 스타터 대체)
/events             약속 목록 (다가오는 / 지난)
/events/new         약속 생성 (일회성/반복 토글)
/events/[id]        약속 상세 — 집계, 응답 버튼, 시간 변경 제안/확정
/profile            닉네임 설정
```

## 6.1 모바일 퍼스트 UX 원칙

이 프로젝트의 1차 사용 환경은 모바일 브라우저다. 모든 화면과 컴포넌트는
데스크톱 레이아웃을 축소하는 방식이 아니라, **모바일 단일 컬럼을 기본값으로
설계한 뒤 `sm:`/`md:` 이상에서만 확장**한다.

### 필수 기준

- 기본 레이아웃은 360px 폭에서 가로 스크롤 없이 동작해야 한다.
- 주요 CTA(로그인, 약속 만들기, 참석/불참/미정, 시간 변경 제안)는 모바일에서
  `w-full` 또는 동등한 엄지 조작 가능한 너비를 기본으로 한다.
- 터치 대상은 최소 44px 높이를 확보한다. `size="sm"` 버튼은 모바일 주요 행동에
  기본값으로 쓰지 않고, 보조 액션에만 제한적으로 사용한다.
- 헤더/상단 네비게이션은 긴 텍스트·인증 버튼 때문에 줄바꿈되더라도 콘텐츠를
  가리지 않아야 하며, 모바일에서는 불필요한 장식보다 핵심 이동/행동을 우선한다.
- 카드 목록은 모바일에서 1열 카드 스택을 기본으로 하며, 각 카드는 한 손 조작에
  맞게 내부 여백과 텍스트 줄 수를 조정한다.
- 상세 화면은 “약속 핵심 정보 → 내 응답 → 참석 현황 → 시간 변경” 순서로
  모바일에서 즉시 행동 가능한 구조를 유지한다.
- 폼은 모바일에서 입력 필드·토글·요일 선택지가 세로 흐름으로 읽혀야 하며,
  제출 버튼은 기본 `w-full`이다.
- 제안/이력/주최자 액션 버튼은 모바일에서 2열 이하 또는 세로 스택으로 배치해
  좁은 화면에서 버튼이 과밀해지지 않게 한다.
- iOS Safari/Android Chrome 기준으로 `input[type="datetime-local"]` 사용성이
  깨지지 않는지 확인한다.

### 현재 UI에서 우선 수정할 지점

- `/` 랜딩: 히어로 세로 여백을 모바일에서 줄이고, CTA는 모바일 기본 `w-full`,
  기능 소개 카드는 1열 압축 카드로 유지한다. 푸터의 중복 테마 스위처는 제거하거나
  모바일에서 숨긴다.
- `/events`: 헤더의 제목/설명과 “약속 만들기” 버튼을 모바일에서는 세로 스택으로
  바꾸고 버튼을 `w-full`로 만든다. 목록 카드 간격은 모바일에서 더 조밀하게 한다.
- `EventCard`: 제목·뱃지 영역이 한 줄에 몰리지 않도록 모바일에서는 제목 아래에
  뱃지를 배치하고, 카드 전체를 충분한 터치 영역으로 만든다.
- `/events/[id]`: 주최자 수정/삭제 버튼, 참석 응답 버튼, 시간 변경 확정/거절 버튼을
  모바일 기본 `w-full`/세로 스택으로 바꾼다. 현재 `size="sm"` 중심 버튼은 주요
  액션에는 작다.
- `RosterList`: 모바일 1열은 유지하되 참석/불참/미정 섹션을 더 압축하거나
  요약 숫자를 먼저 보여 긴 명단이 응답 버튼을 밀어내지 않게 한다.
- `EventForm` / `TimeProposalSection`: 카드가 작은 화면에서 좌우 여백 없이 꽉 차지
  않도록 컨테이너 폭·패딩을 모바일 기준으로 재조정하고, 모든 제출 버튼은 모바일
  기본 `w-full`을 유지한다.
